## Rezime sesije — infra/deploy (Vercel + Railway)

Cela sesija je bila isključivo infra/deploy rad. Nijedan izvorni kod aplikacije nije menjan — sve je rađeno preko Vercel/Railway CLI i MCP alata.

---

### Šta je urađeno

1. **Napravljen `elokal-demo-admin` Vercel projekat.**
   Demo ranije nije imao admin deploy (samo sloj). Root dir `admin/`, env `NUXT_PUBLIC_API_BASE` pokazuje na demo API. Demo sada ima pun stack kao i sloj.

2. **Preimenovani Vercel admin projekti radi konzistentnog naminga.**
   `sloj-admin` → `elokal-sloj-admin`, `demo-admin` → `elokal-demo-admin`. Rename ne premešta auto `.vercel.app` domen, pa su čisti domeni dodati eksplicitno (`vercel domains add`). Finalni frontend URL-ovi (svi vracaju 200): `elokal-demo-web`, `elokal-demo-admin`, `elokal-sloj-web`, `elokal-sloj-admin` (sve `.vercel.app`).

3. **Razjašnjena arhitektura multi-klijent deploya.**
   Admin je jedan deljeni codebase (`admin/`); per-klijent se razlikuje samo build-env `NUXT_PUBLIC_API_BASE`. Svaki klijent ima svoj Railway API servis i svoju Neon bazu. Demo i sloj su potpuno izolovani.

4. **Ispravljena ranija greška: sloj ne deli "glavnu" bazu — glavne baze i nema.**
   Railway projekat `elokal` ima samo 2 servisa (`elokal-demo-api`, `elokal-sloj-api`). Sloj-ov servis je samo imao legacy domen `elokal-api-production` (otud zabuna). Izolacija potvrđena: zasebne Neon baze `elokal-demo-db` i `elokal-sloj-db`, različit admin `created_at` (sloj 2026-05-03, demo 2026-06-04).

5. **Preimenovan sloj Railway API domen (glavna stavka).**
   `elokal-api-production` → `elokal-sloj-api-production.up.railway.app` metodom `railway domain update <stari> --domain elokal-sloj-api-production` — atomični rename u mestu (isti domain ID `5fa1f482`, bez delete+create rizika). Zatim ažuriran `NUXT_PUBLIC_API_BASE` na `elokal-sloj-web` i `elokal-sloj-admin` na novi domen, oba redeploy-ovana. Verifikovano: novi domen vraca 200, stari 404, oba frontenda gađaju novi API. Sloj API bio nedostupan ~par minuta tokom prelaza (prihvaćen kratak prekid).

6. **Root-cause rešen za ponavljajući "moram da pravim Railway token".**
   Railway CLI nije bio trajno ulogovan (`user` blok u `~/.railway/config.json` bio sav null); korisnik je dodavao `RAILWAY_API_TOKEN` inline po komandi (efemerno). Urađen `railway login` → trajna sesija (accessToken + refreshToken, auto-refresh), nalog-wide za sve projekte. Više ne treba praviti novi token.
